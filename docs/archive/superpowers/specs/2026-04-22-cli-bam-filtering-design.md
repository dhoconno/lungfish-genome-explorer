# CLI BAM Filtering Design

## Summary

Add a first-class CLI-backed BAM workflow to Lungfish so BAM filtering is available both from the GUI Inspector and from scripted command-line workflows. The canonical executable will be `lungfish-cli`, with `lungfish` retained as a compatibility alias. The canonical BAM command family will become:

- `lungfish-cli bam filter`
- `lungfish-cli bam markdup`

The existing top-level `lungfish-cli markdup` command remains supported as a compatibility alias.

The filtering workflow will be bundle-centric. It will accept either a reference bundle target or a mapping analysis target, resolve exactly one source alignment track, derive a new sibling filtered BAM track, attach it into the resolved bundle, and record explicit provenance. Mapping-result targets will write only into the copied viewer bundle, not into the top-level analysis directory.

## Goals

- Make BAM filtering available for scripted workflows that chain mapping, filtering, and variant calling.
- Keep GUI and CLI BAM derivation behavior identical by using one shared workflow engine.
- Support both direct bundle targets and mapping-result targets generated by `lungfish-cli map`.
- Require explicit, reproducible CLI inputs for scripting: one source track per invocation and explicit output naming.
- Preserve source BAM tracks and create sibling derived tracks under `alignments/filtered/`.
- Record provenance for every derived BAM.
- Verify the workflow with real artifacts, including at least one real end-to-end CLI execution using actual `samtools`.

## Non-Goals

- Region-scoped BAM filtering in v1.
- Creating new top-level mapping-result artifacts for filtered BAMs.
- Filtering multiple source alignment tracks in one invocation.
- Guessing a source alignment track when multiple tracks are present.
- Replacing the GUI-first workflow with a CLI-only path.

## User-Facing Command Model

### Executable Naming

Canonical executable:

- `lungfish-cli`

Compatibility alias:

- `lungfish`

All new docs, examples, help text, and tests should prefer `lungfish-cli`. The shorter `lungfish` name remains available for compatibility, but it should stop being the primary documented surface.

### BAM Command Family

Canonical commands:

- `lungfish-cli bam filter`
- `lungfish-cli bam markdup`

Compatibility alias:

- `lungfish-cli markdup`

The `bam` family becomes the discoverable home for BAM-specific operations. The top-level `markdup` command is retained to avoid breaking existing scripts.

### `bam filter` Surface

V1 command shape:

```bash
lungfish-cli bam filter \
  --bundle /path/to/ref.lungfishref \
  --alignment-track source-track-id \
  --output-track-name "Exact Match Reads" \
  --mapped-only \
  --exact-match
```

```bash
lungfish-cli bam filter \
  --mapping-result /path/to/Analyses/mapping-2026-04-22 \
  --alignment-track mapped-reads \
  --output-track-name "Deduplicated High Identity" \
  --remove-duplicates \
  --min-percent-identity 99.5
```

Required target selection:

- `--bundle <path>` or `--mapping-result <path>`
- these options are mutually exclusive

Required track selection:

- `--alignment-track <id>`
- exactly one source BAM track per invocation

Required output naming:

- `--output-track-name <name>`

Composable v1 filter flags:

- `--mapped-only`
- `--primary-only`
- `--min-mapq <int>`
- `--exclude-marked-duplicates`
- `--remove-duplicates`
- `--exact-match`
- `--min-percent-identity <double>`

Deferred from v1:

- `--region`
- multi-track invocation
- preset-based filtering

## Target Resolution

### Bundle Targets

When `--bundle` is provided:

- open the `.lungfishref` bundle directly
- resolve the requested alignment track from that bundle
- attach the derived BAM back into that same bundle

### Mapping Result Targets

When `--mapping-result` is provided:

- load the mapping result metadata from the analysis directory
- resolve `viewerBundleURL`
- fail clearly if no viewer bundle exists
- resolve the requested alignment track from the viewer bundle
- attach the derived BAM as a sibling track inside that copied viewer bundle only

This workflow must not create a new top-level mapping analysis artifact. The filtered BAM exists only as a sibling alignment track inside the resolved viewer bundle.

## Source Track Rules

The CLI operates on one source alignment track per invocation.

This is explicit and never guessed:

- `--alignment-track` is always required
- the command fails if the track does not exist
- the command fails if the track is not BAM-backed

A BAM spanning multiple chromosomes or contigs is still one source track. Whole-track filtering applies across the entire BAM unless a future version adds region filtering.

## Filter Semantics

### General Filters

- `--mapped-only` excludes unmapped reads
- `--primary-only` excludes secondary and supplementary alignments
- `--min-mapq <int>` applies a mapping-quality threshold

### Duplicate Filters

Two duplicate-related behaviors must exist because they have different meaning and runtime cost:

- `--exclude-marked-duplicates`
  - filters out reads already marked with duplicate flag `0x400`
  - does not run duplicate detection

- `--remove-duplicates`
  - actively runs duplicate marking as preprocessing
  - removes duplicates from that derived intermediate

These flags are mutually exclusive.

### Identity Filters

- `--exact-match`
  - retains alignments where `NM == 0`

- `--min-percent-identity <double>`
  - retains alignments meeting the minimum identity threshold

These flags are mutually exclusive.

Identity filtering requires `NM` tags. The command must preflight required tags and fail clearly if the source BAM lacks them.

## Architecture

### Shared Workflow Engine

The BAM derivation engine must move out of `LungfishApp` into a shared non-UI layer so both the GUI Inspector and the CLI use the same implementation. `LungfishWorkflow` is the preferred home because it already owns managed workflow execution and mapping-result types.

The shared layer will own:

- target resolution for bundle and mapping-result inputs
- filter request models
- `samtools view` command planning
- duplicate preprocessing via markdup
- bundle-track attachment/import
- provenance writing
- validation and failure messages

### Layer Responsibilities

`LungfishWorkflow`:

- shared BAM filter request/result types
- shared target resolver
- shared BAM derivation workflow service
- shared markdup preprocessing integration
- provenance and attachment logic

`LungfishCLI`:

- argument parsing
- user-facing validation
- text and JSON event emission
- compatibility alias routing

`LungfishApp`:

- Inspector state and UI
- thin invocation of the shared workflow service

This split ensures that GUI and CLI semantics cannot drift apart.

## Execution Flow

The `lungfish-cli bam filter` workflow is deterministic and bundle-first:

1. Parse and validate CLI arguments.
2. Resolve exactly one target bundle:
   - direct bundle, or
   - mapping result resolved to its viewer bundle.
3. Resolve exactly one source alignment track.
4. Preflight:
   - conflicting flags
   - missing target or track
   - missing viewer bundle for mapping results
   - missing required tags such as `NM`
5. Start a bundle-scoped operation lock.
6. Run optional duplicate preprocessing for `--remove-duplicates`.
7. Run `samtools view`.
8. Run `samtools sort`.
9. Run `samtools index`.
10. Attach/import the derived BAM into the resolved bundle under `alignments/filtered/`.
11. Write derivation metadata and provenance.
12. Emit final text or JSON result.

The source BAM track is never modified in place.

## Provenance Requirements

Every derived BAM must record:

- source bundle path
- source alignment track ID and name
- whether the target came from a mapping result
- mapping-result path when applicable
- created alignment track ID and name
- created BAM/index/metadata DB paths
- full filter settings
- whether duplicate preprocessing ran
- command history in execution order
- preflight warnings or tag-validation failures when relevant

The CLI and GUI must show the same underlying provenance data because they are writing through the same workflow engine.

## CLI Output Model

### Default Output

Human-readable progress and completion text.

### JSON Output

`--output-format json` emits structured events suitable for scripting and wrappers.

At minimum, final success output should include:

- resolved bundle path
- source track ID
- created track ID
- created track name
- created BAM path
- created BAI path
- created metadata DB path
- target kind (`bundle` or `mapping-result`)
- mapping result path when applicable

Structured failures should include a stable error code and message.

## `bam markdup` Compatibility

Canonical command:

- `lungfish-cli bam markdup`

Compatibility alias:

- `lungfish-cli markdup`

Both entry points should call the same implementation. Tests should verify equivalent behavior so the compatibility path remains trustworthy during migration.

## Testing Strategy

This work must be tested with real artifacts, not only stubs.

Required coverage:

- unit tests for CLI argument validation and request construction
- shared-service tests for target resolution, duplicate semantics, tag validation, and provenance writing
- CLI tests for text output mode
- CLI tests for JSON output mode
- compatibility tests for `bam markdup` and top-level `markdup`
- mapping-result resolution tests covering `viewerBundleURL`

Required real-artifact validation:

- at least one real end-to-end CLI test or harness execution using an actual BAM fixture and actual `samtools`
- this test must exercise the real filtering command path, not only a mocked runner
- it must verify the output bundle contents and the derived track’s provenance/metadata
- it must cover one direct bundle target
- it must cover one mapping-result target, either in the same test suite or as a second integration case

This requirement is explicit because the current app-backed tests rely heavily on recording and stub runners, which are valuable but not sufficient for CLI workflow confidence.

## Migration Plan

1. Standardize the executable/docs direction around `lungfish-cli`, keeping `lungfish` as an alias.
2. Move the BAM filtering workflow from app-owned code into a shared non-UI layer.
3. Rewire the Inspector to use the shared workflow service.
4. Add `lungfish-cli bam filter`.
5. Add `lungfish-cli bam markdup` as canonical while preserving `lungfish-cli markdup`.
6. Add real-artifact end-to-end CLI validation.
7. Update help text and docs to prefer `lungfish-cli`.

This ordering keeps GUI behavior stable while adding the CLI surface needed for scripting.

## Accepted Decisions

- canonical executable is `lungfish-cli`
- `lungfish` remains as an alias
- top-level `bam` family is the canonical home for BAM operations
- one source alignment track per invocation
- `--alignment-track` is explicit, not guessed
- `--output-track-name` is required
- composable filter flags are used in v1
- region-scoped filtering is deferred
- mapping-result targets resolve to the copied viewer bundle only
- both duplicate behaviors exist:
  - exclude already marked duplicates
  - actively run duplicate removal
- real-artifact end-to-end testing is required
